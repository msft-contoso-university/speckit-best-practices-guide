What is Spec-Kit?

Spec-Kit is a Specification-Driven Development (SDD) tool from GitHub that inverts the traditional code-first approach. Specifications become the primary artifact—code is the generated output that serves the specification.

📝 Intent-Driven

Specs capture WHAT you want, not HOW to build it. AI handles implementation details based on structured, precise specifications.

⚙️ Executable Specs

Specifications are living instructions that drive multi-step development workflows, enable AI agents to build iteratively, and provide feedback loops.

🔄 Multi-Phase

Supports Greenfield (0-to-1), Creative Exploration (prototyping), and Brownfield (refactoring) development phases.

Key Insight: In SDD, maintaining software means evolving specifications. Debugging means fixing specifications that generate incorrect code. The spec is truth—code is its expression.

The .specify/ Directory Structure

# Every Spec-Kit project creates this structure: .specify/ ├── constitution.md # Core principles and rules (the "OS" of your project) ├── specification.md # Technical requirements (WHAT to build) ├── plan.md # Implementation roadmap (HOW + tech stack) └── tasks.md # Actionable task list (executable steps)

Build it

1

/speckit.constitution — Establish Project Rules

Creates the governing principles. Should define: tech constraints, quality standards, testing requirements, approved libraries. Think of it as your project's "operating system."

2

/speckit.specify — Define Requirements

Describe WHAT you want to build with user stories, acceptance criteria, and success metrics. Avoid mentioning tech stack here—focus on intent. Automatically creates feature branches and structured directories.

3

/speckit.clarify — Identify Gaps (Optional but Recommended)

AI identifies underspecified areas before planning begins. Produces [NEEDS CLARIFICATION] markers that must be resolved. This saves massive token waste in later steps.

4

/speckit.plan — Technical Implementation Plan

Now introduce tech stack, architecture, and phased implementation details. Generates supporting docs: data models, API contracts, research docs, and quickstart guides.

5

/speckit.tasks — Generate Task List

Converts the plan into atomic, executable tasks (1-4 hours each). Marks independent tasks for parallelization. Creates the actionable backlog.

6

/speckit.implement — Execute

AI works through all tasks systematically, building the application according to constitution, specification, and plan. This is where code generation happens.

Observer Watch: If the developer jumps straight to /speckit.implement without constitution + specify + plan, flag it immediately. This produces low-quality, unfocused output and wastes tokens on regeneration.

Constitution Best Practices

The constitution is the highest-impact artifact. A strong constitution constrains the LLM to produce architecturally consistent, high-quality code. A weak one leads to drift and wasted iterations.

✅ DO

  • Write specific, testable principles ("All API endpoints must return proper HTTP status codes")
  • Include approved library lists with rationale
  • Define what "done" looks like for a feature
  • Reference specific patterns by name
  • Keep it concise—LLMs process shorter docs more accurately
  • Update when you discover new patterns

❌ DON'T

  • Write vague principles ("Code should be clean")
  • Include 50+ linting rules (use .eslintrc instead)
  • Copy generic best practices that don't fit your project
  • Lock into decisions too early—constitutions can evolve
  • Forget to document WHY a principle exists
  • Make it so long the LLM truncates it from context

Example: Well-Structured Constitution Article

## Article I: Library-First Development When a task can be solved with a well-maintained library, we MUST use the library. Why: Implementations using libraries (rss-parser, cheerio) achieved 95% spec adherence vs 40% for custom parsers in validation testing. Approved Libraries: - RSS parsing: rss-parser - Web scraping: cheerio - Database: better-sqlite3 - Testing: vitest

Token Impact: A well-written constitution of 200-400 words outperforms a 2000-word one. The LLM reads the full constitution on every /speckit.constitution call. Keep it dense and actionable.

Token Optimization Strategies

🎯 Front-Load Precision

  • Invest time in clear, unambiguous specs upfront
  • Every [NEEDS CLARIFICATION] marker resolved before planning saves 3-5x tokens downstream
  • Use /speckit.clarify before /speckit.plan
  • Ambiguity in specs = multiple regeneration cycles during implementation

📊 Hierarchical Detail Management

  • Keep plan.md high-level and readable
  • Move code samples and algorithms to implementation-details/ subdirectory
  • The LLM loads full file contents—bloated files waste context window
  • Use separate files for data models, API contracts, and research

✂️ Atomic Tasks

  • Each task in tasks.md should be completable in 1-4 hours
  • Include: Input, Output, Error handling, Test criteria
  • Vague tasks like "Build the RSS system" cause the LLM to hallucinate scope
  • Atomic tasks = fewer retries = fewer tokens

🔁 Prevent Regeneration Loops

  • Use /speckit.analyze before /speckit.implement to catch misalignment
  • Validate constitution + spec + plan consistency first
  • Fix spec issues BEFORE implementation, never during
  • A single spec change mid-implement can waste 50%+ of spent tokens

Red Flag: If the developer is repeatedly asking the AI to "redo" or "try again" during implementation, the upstream specs are likely unclear. Stop, go back to /speckit.specify or /speckit.clarify, fix the ambiguity, then resume. This is the #1 token waste pattern.

GitHub Copilot Best Practices with Spec-Kit

Spec-Kit is designed to work with GitHub Copilot's Agent Mode. These practices ensure maximum effectiveness and minimal token waste when using them together.

🤖 Agent Mode Configuration

  • Initialize with --ai copilot: specify init my-project --ai copilot
  • This configures slash commands for Copilot's interface
  • Agent Mode processes the full .specify/ directory as context
  • Copilot's context window benefits from concise, well-structured specs

📁 Context Management

  • Keep .specify/ in workspace root—Copilot auto-indexes it
  • Use .github/copilot-instructions.md to point Copilot at your constitution
  • Smaller, focused files > large monolithic specs (aids retrieval)
  • Pin critical spec files in your editor for instant context

Copilot Token Efficiency Patterns

💬 Prompt Hygiene

  • Be specific in requests: "Implement task 2.3 from tasks.md" not "build the next thing"
  • Reference spec artifacts by name to anchor the LLM
  • One intent per message—compound requests confuse context
  • Use /speckit.tasks to remind Copilot of current work state

🧹 Session Management

  • Start new chat sessions for new features to reset context
  • Long conversations accumulate noise that degrades output
  • Use /speckit.constitution at session start to ground the model
  • After 15-20 messages, consider a fresh session with clear intent

🔒 Scope Control

  • Close unrelated files before implementation (reduces context noise)
  • Use #file references to direct Copilot to specific specs
  • Prefer @workspace for broad questions, direct references for implementation
  • Exclude node_modules/, dist/ from Copilot's indexing

Common Pitfalls to Watch For

As an observer, these are the patterns that lead to poor outcomes. Flag them early to save time and tokens.

🚨 1. Skipping the Constitution

Jumping to /speckit.specify without establishing principles first. The constitution constrains the LLM—without it, outputs are generic and lack architectural consistency.

Fix: Always start with /speckit.constitution. Even 5 minutes of principle-setting saves hours of rework.

🚨 2. Over-Specification

Writing 2000+ word constitutions with linting rules, style guides, and micro-decisions. The LLM truncates or loses focus with too much context.

Fix: Constitutional principles should be architectural, not stylistic. Use tooling (.eslintrc, .prettierrc) for style.

🚨 3. Mixing WHAT and HOW in Specs

Including tech stack decisions in specification.md. This couples requirements to implementation, making specs brittle and hard to iterate.

Fix: Spec = "Users can drag-and-drop albums to reorder." Plan = "Use React DnD with Zustand state management."

🚨 4. Not Using Clarify Before Plan

Going directly from /speckit.specify to /speckit.plan. Unresolved ambiguities propagate into the plan and multiply during implementation.

Fix: Always run /speckit.clarify. Resolve every [NEEDS CLARIFICATION] marker before proceeding.

🚨 5. Long Conversations Without Re-grounding

15+ message chains without invoking /speckit.constitution. Context drifts, the LLM "forgets" principles, and output quality degrades progressively.

Fix: Re-invoke /speckit.constitution every 5-10 messages. Start fresh sessions for new features.

🚨 6. Vague Tasks Leading to Token Waste

Tasks like "Build the authentication system" instead of atomic tasks with clear input/output/test criteria. Causes the LLM to hallucinate scope.

Fix: Each task must have: Input, Output, Error handling strategy, and Test criteria. Target 1-4 hour completion time.

🚨 7. Not Version-Controlling Specs

Treating .specify/ as ephemeral. When specs evolve without history, you lose the ability to correlate spec changes with quality outcomes.

Fix: git add .specify/ and commit with every change. Use branches for experimental spec changes.

🚨 8. Changing Specs Mid-Implementation

Modifying specification.md or plan.md while /speckit.implement is in progress. This creates inconsistencies and forces expensive re-generation.

Fix: Complete current implementation phase, then update specs and regenerate from the plan phase down.

GitHub Copilot Token Patterns

Specific behaviors in GitHub Copilot that impact token efficiency when combined with Spec-Kit workflows.

📥 Context Window Loading

Copilot loads open files, referenced files, and .github/copilot-instructions.md into context. Every additional file consumes tokens from the response budget.

Action: Close irrelevant files. Keep only spec files + current implementation target open.

📝 Instruction File Size

The copilot-instructions.md file is loaded on every interaction. If it's 1000+ words, you're spending tokens before the user even types.

Action: Keep instructions under 300 words. Point to spec files rather than duplicating content.

🔄 Conversation History

Each message in a session carries forward. After 15-20 exchanges, early context gets compressed or dropped. Spec drift accelerates.

Action: Start fresh sessions per feature. Use /speckit.constitution to re-anchor after long exchanges.

🎯 #file References

Using #file:path/to/file in Copilot Chat explicitly loads that file into context. This is a precision tool for directing attention.

Action: Reference #file:.specify/constitution.md when you need principle alignment. More targeted than @workspace.

🚫 @workspace Scope

@workspace triggers a search across all indexed files. Useful for discovery, but expensive for implementation where you already know the context.

Action: Use @workspace during specify/plan phases. Switch to direct #file references during implement.

⚡ Agent Mode Multi-File

Agent Mode can read and write multiple files per turn. Each file operation adds to token usage. Unfocused agent actions multiply costs.

Action: Give Agent Mode precise tasks with explicit file targets. "Create src/db/schema.ts per task 2.1" beats "set up the database."

Golden Rule: Precision in prompts = fewer tokens spent. "Implement task 3.2 from tasks.md using better-sqlite3 per constitution Article I" consumes far fewer total tokens than "build the database thing" followed by 5 correction messages.

Cavekit's Backprop Mechanism

The killer feature Spec-Kit doesn't have: automatic learning from failures. Every test failure becomes a permanent invariant the spec never forgets.

How Backprop Works

  1. Test fails during /ck:build
  2. Agent diagnoses root cause
  3. Adds §B entry: B3|2026-05-19|race on write|V3
  4. If no invariant covers it, creates new §V entry
  5. Invariant prevents recurrence across all future sessions
  6. SPEC.md grows smarter with every failure

Why This Matters for Token Billing

  • Prevents repeated mistakes—each bug class caught once, permanently
  • Reduces regeneration cycles—invariants prevent the same failure pattern
  • Self-improving spec—SPEC.md gets better without manual effort
  • Audit trail—§B table shows what went wrong and when
  • Token ROI—a 10-token invariant can prevent 1000+ tokens of failed implementation

Observer Checklist

Use this checklist as you watch the developer scaffold their application. Each item represents a best practice checkpoint.

🏗️ Setup & Initialization

  • Spec-Kit installed and specify check passes
  • Project initialized with specify init --ai copilot
  • .specify/ directory committed to git
  • .github/copilot-instructions.md references spec files
  • Irrelevant files closed in editor

📐 Constitution Quality

  • Constitution created before specification
  • Principles are specific and testable (not vague)
  • Includes approved library list with rationale
  • Under 500 words (concise and dense)
  • Defines what "done" looks like
  • Documents WHY each principle exists

📝 Specification Quality

  • Focuses on WHAT and WHY, not HOW
  • Has measurable acceptance criteria
  • No implementation details (tech stack, libraries)
  • All [NEEDS CLARIFICATION] markers resolved
  • User stories have clear success criteria

⚡ Token Efficiency

  • /speckit.clarify used before /speckit.plan
  • /speckit.constitution invoked every 5-10 messages
  • New chat sessions started for new features
  • Prompts reference specific tasks by number
  • No compound requests (one intent per message)
  • Not repeatedly asking AI to "redo" or "try again"
  • Agent Mode given explicit file targets
  • #file references used instead of broad @workspace during implement

🚀 Implementation Quality

  • Tasks are atomic (1-4 hours, clear I/O/test)
  • /speckit.analyze run before /speckit.implement
  • No spec changes during active implementation
  • Plan uses hierarchical detail (sub-files for complexity)
  • Specs version-controlled with meaningful commits
  • Phase gates checked before proceeding

🛡️ Drift Prevention

  • AI suggestions checked against constitution
  • Tech choices align with plan.md (no ad-hoc introductions)
  • Features traced back to specification.md
  • No "nice to have" features added without spec update

🪨 Caveman (Optional Token Savings)

  • /caveman activated at session start (lite/full/ultra)
  • Compression level appropriate for dev experience
  • /caveman-stats checked periodically to measure savings
  • /caveman-compress applied to stable spec files for input savings
  • Code output verified as byte-identical (compression = narration only)
